## 一、社交可供性设计 (Social Affordances)

### 什么是可供性？

**可供性 (Affordance)** 是用户与对象之间互动的可能性。

**设计属性**：
- **实用性**: 设计者决定什么是有用的（如门把手放在手可触及的高度）
- **符号性**: 文化上可理解的形状（如门把手的"可转动"形状）

**数字世界的独特优势**：设计者对可供性有完全的控制权。在现实世界中无法控制门的"可踢性"，但在数字世界中可以简单地不创建"踢"这个可供性。

### 社交可供性

**社交可供性**是游戏中创建用户与用户之间互动可能性的 UX 元素。

## 二、封闭可供性 vs 开放可供性

### 封闭可供性 (Closed Affordances)

**定义**：对互动的绝对限制，精确定义互动并排除意外的边缘情况。

**特点**：
- 移除大范围的问题行为
- 难以或不可能用于伤害他人
- 减少玩家代理权和表达力

**案例**：
- **Journey**: 玩家只能发出蜂鸣声，没有其他符号交流方式
- **移动游戏**: 消除开放文字聊天、手势、语音聊天，甚至避免玩家相遇

**价值**：减少恶意破坏和后续的审核成本

**代价**：减少玩家代理权和表达力，消除人类互动的好的一面

### 开放可供性 (Open Affordances)

**定义**：创建社交机会，允许多个自由度和社会秩序的形成。

**特点**：
- 用户可以玩社会秩序
- 可以形成复杂的社交结构
- 人类会自然地撬开设计者试图关闭的沟通渠道

**风险**：开放可供性自然会产生意料之外的边缘情况，包括有毒行为

### 最佳实践：渐进式开放

**核心原则**：低信任玩家受益于封闭可供性，高信任玩家需要开放可供性

**设计策略**：
1. 从封闭可供性开始，提供相对安全的空间
2. 随着信任增长，设计选择加入的门
3. 建立进展系统，随玩家信任增长逐步解锁更开放的社交行动

**案例: Apex Legends**：
- 非常有限的情境 ping 系统
- 难以组合成复杂句子
- 在保持沟通能力的同时限制了毒性

## 三、信件圈 (The Letter Circle)

### 设计目标

设计一种值得使用的社交媒体形式：
- 帮助创建和管理专注于深思熟虑写作和对话的小型社交群体
- 基于社交系统设计原则，创造长期生存的社区

### 历史灵感：书信交流

18-19 世纪：
- 艺术家和作家交换长信，分享最新挣扎
- 科学家交换问题和理论笔记
- 许多变革性的知识分子物受书信写作影响

**价值**：
- 被迫用长文形式解释自己具有独特的生成性
- 同行用心和理解回应具有治愈和启发作用

### 基本结构

| 元素 | 说明 |
|------|------|
| **群体规模** | 最多 50 人（Dunbar 数字的合理子集） |
| **入组要求** | 必须提交一封相关主题的信件（3 个月窗口） |
| **维持要求** | 每年至少 1 封长信 + 2 封回复 |
| **自动化管理** | 未达标自动移除 |

### 什么是信件？

**定义**：发送给你尊重并希望交流的人的深思熟虑的写作（1-6 页）

**内容可能包含**：
- 开始对话的观察
- 正在解决的问题
- 有帮助的认知工具、模型或视角
- 生活事件描述（建立亲密和信任）
- 对先前信件的深思熟虑的回应

**信件不是**：
- 完整的研究或白皮书
- 短推文
- 广播给所有人的内容

### 什么是回复？

**定义**：讨论信件要点的较短文字，尊重和协作性质

**好回复包含**：
- 对主题的替代视角
- 支持或增加细微差别的例子
- 基于主题的发挥

**回复不是**：
- 简短评论（"好！" "错了！"）
- 离题内容

### 自动化群体管理

| 功能 | 说明 |
|------|------|
| **提醒邮件** | 定期提醒未发信/回复的成员 |
| **自动移除** | 未达标成员自动移除，减少戏剧性 |
| **招募提醒** | 成员少于 50 人时自动提醒邀请 |

### 50 人限制的原因

**为什么不是更大？**
- 超过这个数字，人们倾向于闭嘴
- 不知道群里有什么人，变得谨慎
- 信息太多，人们开始忽略

**为什么不是更小？**
- 可能没有足够的互动量
- 可能没有人及时回应
- 可能忘记群体的存在

## 四、设计模式提炼

### 社交可供性设计模式

| 模式 | 说明 | 应用 |
|------|------|------|
| **封闭启动** | 新用户从受限的社交互动开始 | Journey 的蜂鸣声 |
| **渐进解锁** | 随信任增加开放更多社交功能 | Sky 的友谊系统 |
| **情境限制** | 使用预设的情境响应而非自由文本 | Apex Legends 的 ping |
| **选择加入门** | 用户主动选择进入更开放的社交空间 | 分级好友系统 |

### 社交系统自动化模式

| 模式 | 说明 |
|------|------|
| **贡献门槛** | 新用户必须先贡献才能加入（信件圈的首次信件） |
| **维持要求** | 定期贡献以保持成员资格 |
| **自动清理** | 未达标自动移除，减少人为戏剧 |
| **招募循环** | 成员负责邀请新成员维持群体活力 |

### 互惠设计模式

| 模式 | 说明 |
|------|------|
| **强制回应** | 写作者必须得到回应（回复要求） |
| **可见贡献** | 所有成员都能看到贡献 |
| **社交契约** | 前置明确参与规则 |

## 五、核心洞察

### 社交可供性洞察

1. **设计者有完全控制权**: 数字世界中可以选择不创建某种可供性
2. **封闭 ≠ 禁止开放**: 而是从封闭开始，渐进开放
3. **人类会撬开封闭系统**: 如果需求存在，社区会找到绕过限制的方法
4. **信任是开放的前提**: 低信任需要封闭，高信任需要开放

### 社交系统设计洞察

1. **群体规模很关键**: 50 人左右是自管理群体形成有意义关系的上限
2. **主动贡献 > 被动浏览**: 新用户不应只是"潜水"，必须立即贡献
3. **自动化减少戏剧**: 自动移除比人工踢人更少戏剧性
4. **互惠是社区生命线**: 写作者需要知道有人在听
5. **前置社交契约**: 参与规则要在开始时明确

### 商业洞察

1. **不需要规模化**: 信件圈不需要成为 VC 支持的盈利实体
2. **内在动机足够**: 掌握感、连接感、归属感是强大动力
3. **小规模可持续**: 志愿者可以资助托管成本
