## 核心问题

每个星期都有"更好"的 AI 模型，但很多问题没有"更好"的答案，只有"正确"的答案。**"更好"意味着什么？我们如何管理这些系统？**

---

## 两种任务类型

### 类型一：主观质量（"更好"有意义）
- 创意写作
- 图像生成（Midjourney 3 → 4 → 5 → 6.1）
- 营销文案
- **特点**：没有"错误"答案，错误容易发现和修复

### 类型二：二元正确性（只有对错）
- 具体数据查询
- 精确信息检索
- 事实核查
- **特点**：答案不是百分比，而是二元的

### 早期产品市场契合的领域
**编程 + 营销** = 错误容易测试/发现 + 没有绝对正确答案

## 案例研究：电梯操作员

### 问题
"1980 年美国有多少人受雇为电梯操作员？"

### 结果
- **正确答案**：21,982（美国人口普查数据，PDF 第 17 页）
- **ChatGPT 4o 尝试 1**：给出具体但错误的答案，无来源
- **尝试 2**（提供来源）：不同但仍然错误的答案，附有看起来正确的来源链接
- **尝试 3-6**：提供 PDF、指出具体位置、要求浏览网页 → 全部失败

### 核心问题
> "问题不在于数字是错的，而在于我不做所有工作就无法知道它是否正确。"

## 理论框架

### LLM 的本质
- **概率性、统计性系统**：告诉你一个好的答案大概长什么样
- **不是确定性系统**：不告诉你答案是什么
- **不"知道"或"理解"**：只是近似

### 更好的模型 ≠ 正确的答案
- "更好"的模型更接近近似
- 可能在某类问题上明显更好
- 但仍然不是提供"正确"答案

### 未知数
- 这种"理解"是否会随模型扩展自发涌现？
- 像 Zeno 悖论，永远达不到目标但收敛到 99.99%？
- 需要其他理论突破？
- OpenAI O3 的"推理"是解决路径？
- **我们不知道，目前没有理论框架能告诉我们**

## 战略思考

### 1. 当前的工程解决方案
大多数使用生成式 AI 的公司：
- 将 LLM 作为 API 调用包装在传统确定性软件中
- 通过工具、流程、控制和 UX 管理错误率
- 给马套上挽具、戴上眼罩和缰绳

### 2. 未来的架构方向
**方向 A**：在可预测的框架内控制 LLM
**方向 B**：给 LLM 可预测的工具
- LLM 告诉 SAP 运行什么查询
- 概率系统控制确定性系统
- 这是"代理"系统的一种思考方式

### 3. 颠覆理论
> "重要的新技术往往在上一代技术重视的事情上表现糟糕，但它们做了其他重要的事情。"

- 问 LLM 能否做精确信息检索，可能像问 Apple II 能否匹配大型机的正常运行时间
- 问能否在 Netscape 中构建 Photoshop
- **不是重点，不代表没用**
- 它们解锁了其他东西

### 4. 期望的转变
**Motorola 故事的启示**：
- iPod 掉落会坏 → 你怪自己
- 手机掉落会坏 → 你怪制造商
- iPhone 推出后 → 掉落会坏，但你可能怪自己

> "50 年的消费计算训练我们期望计算机是'正确'的——可预测、确定性的系统。但如果你翻转这种期望，你会得到什么回报？"
