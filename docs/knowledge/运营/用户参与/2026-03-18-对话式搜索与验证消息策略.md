## 概述

本文档整理了 OneSignal 博客中关于两个 2025-2026 年核心用户参与趋势的深度洞察：**对话式搜索**（Conversational Search）和**验证消息**（Verified Messaging）。这两个趋势正在重新定义品牌与客户的互动方式。

---

## 第一部分：对话式搜索改变客户参与

> **核心洞察**：传统搜索尝试解读，对话式搜索帮助客户澄清他们真正想要什么。

### 从搜索到询问的转变

| 维度 | 传统搜索 | 对话式搜索 |
|------|----------|------------|
| **交互方式** | 猜测关键词 | 用自然语言询问 |
| **目标** | 尝试解读意图 | 帮助用户澄清需求 |
| **结果** | 猜测匹配 | 精确匹配 |
| **用户体验** | 多次搜索+多个标签页 | 单线程完成 |

### 对话式搜索为何现在发生

#### 1. 人们用自然语言思考
- 在移动端、匆忙或不确定时，用户不想将需求翻译成过滤逻辑
- 示例：家长在出租车上输入"给9岁喜欢科学的孩子送礼物，30美元以内，周五送到"，立即获得精选选项，而非847个结果的分类页面

#### 2. 期望从结果转向解决
- 客户不想要更多链接，他们想要下一步显而易见
- 示例：旅行者问"这家酒店有延迟入住和停车位吗？"，助手回答、确认政策并提供预订

#### 3. 对话比导航适应更快
- 菜单需要预测路径，对话发现路径
- 示例：用户从"旅行降噪耳机"开始，两个问题后意识到他们实际上需要30+小时电池续航、可折叠设计和长航班舒适感

#### 4. 信任需要呼吸空间
- 购买前的"等等..."时刻充满疑问
- 对话让客户无摩擦地提问
- 示例：购物者想知道如果不合身是否可退货，助手在怀疑出现时立即展示退货政策和尺码指南

### 对话式发现实践案例

| 行业 | 场景 | 对话效果 |
|------|------|----------|
| **零售** | 跑鞋推荐 | 推荐三个选项、解释稳定性特征、展示类似跑者评论、在聊天中完成结账 |
| **旅行/本地服务** | 牙医预约 | 找到办公室附近、讲英语、有晚间时段的牙医，查看可用时段并预订 |
| **金融科技** | 套餐比较 | 用简单语言比较两个套餐、解释权衡、根据用户习惯估算月度节省 |
| **电信支持** | 数据问题 | 突出消耗数据的内容、建议与实际使用一致的升级、立即激活 |
| **医疗信息** | 孕前检查 | 提供清单、简要说明、典型时间、连接附近实验室的预订流程 |
| **B2B软件** | 生命周期消息 | 提供相关用例、实施时间线、类似公司证明点、复杂需求专家对接 |

### 新漏斗：从点击到完成

```
传统漏斗优化             →        对话式漏斗优化
流量和点击                        完成

Ask（询问）→ Clarify（澄清）→ Recommend（推荐）→ Prove（证明）→ Act（行动）
    ↓              ↓                ↓               ↓            ↓
 用户用自然语言   几个轻量级问题    排名选项+清晰    怀疑时刻出现   在线程内
 陈述需求        揭示约束         推理           证据           完成行动
```

#### 五阶段详解

| 阶段 | 描述 | 设计要点 |
|------|------|----------|
| **Ask** | 客户用自然语言陈述需求 | 无需关键词翻译 |
| **Clarify** | 轻量级问题揭示约束 | 问题感觉有帮助，非审问式 |
| **Recommend** | 排名选项+清晰推理 | 透明度建立信任 |
| **Prove** | 怀疑时刻出现证据 | 验证评论、退货政策、配送确认 |
| **Act** | 在线程内完成下一步 | 每增加一个导航点都降低完成率 |

### 客户参与团队需要的新能力

| 能力 | 说明 |
|------|------|
| **对话设计成为增长杠杆** | 时机、语气、问题顺序影响结果 |
| **实时编排** | 答案不够，线程应完成行动 |
| **零方数据更有价值** | 客户在体验有帮助时主动提供偏好 |
| **测量需要升级** | 追踪完成率、解决时间、转化、满意度 |
| **信任与合规** | 清晰披露、最小数据收集、安全流程、易用人工切换 |

### RCS 的独特定位

| 特点 | 价值 |
|------|------|
| **富交互** | 按钮、建议回复、轮播、媒体减少摩擦 |
| **持久线程** | 支持异步生活，用户可暂停返回 |
| **消息信号高意图** | 开始对话是故意行为，非浏览 |
| **运营回报明确** | 更少步骤=更少流失 |

### 90天准备计划

```
Weeks 1-2:  发现最高发现问题
            └── 使用站内搜索日志、支持工单、销售记录
            └── 寻找模式："我需要哪个套餐？""这能与我的设备配合吗？"

Weeks 3-4:  选择3个高意图旅程
            └── 产品查找器、预订、升级、故障排除
            └── 专注于高摩擦时刻

Weeks 5-6:  构建对话架构
            └── 定义意图、实体、约束、回退
            └── 映射"不臃肿"在你的语境中意味着什么

Weeks 7-8:  设定信任边界
            └── 决定什么可以自动化、什么需要验证、什么需要人工
            └── 账单更改需要身份验证，运输问题不需要

Weeks 9-10: 启动并测量行为
            └── 追踪完成、流失点、满意度、转化
            └── 如果大多数流失发生在配送日期问题，提前移动配送保证

Weeks 11-12: 在对话中添加信任信号
            └── 在结账犹豫时，在同一线程中展示政策、验证、评论、保证
```

## 第二部分：信任是新的增长渠道

> **核心洞察**：身份验证现在成为产品体验的一部分。

### 验证消息成为新基线

#### 三大原因

| 原因 | 说明 | 案例 |
|------|------|------|
| **垃圾信息改变默认行为** | 收件箱不仅拥挤，还有风险 | 医疗预约提醒：患者犹豫未知发送者，导致错过访问 |
| **冒充成本低且可扩展** | 假冒品牌消息几分钟创建，几乎零成本 | 银行客户收到随机号码的"可疑登录"短信，合法欺诈警报也开始被忽略 |
| **期望翻转** | 人们现在期望品牌证明身份 | 旅行公司发送最后一分钟登机口变更，客户不想"再次确认" |

### 验证如何改变表现

```
验证序列：
┌─────────────────────────────────────────────────────────────────┐
│  注意力提升        │  身份清晰时犹豫下降                         │
│      ↓             │                                            │
│  参与度上升        │  无"这是真的吗？"时刻=更多点击、回复、选择加入│
│      ↓             │                                            │
│  信号变好          │  客户参与时获得可使用的数据                 │
│      ↓             │                                            │
│  编排更智能        │  更好信号=更好的序列和渠道选择              │
│      ↓             │                                            │
│  增长显现          │  转化提升、复购上升、流失下降               │
└─────────────────────────────────────────────────────────────────┘
```

### 验证消息实际意味着什么

验证是消息体验内的发送者身份证明，不是"点击这里验证"，不是单独步骤。

#### 客户注意到的三要素

| 要素 | 说明 |
|------|------|
| **清晰品牌存在** | 名称和品牌信号立即可见 |
| **一致身份** | 每次关键时刻都显示相同的验证存在 |
| **可预测体验** | 客户学习你的消息外观和安全行为 |

> **关键洞察**：验证不取代好的消息，它解锁好的消息。优秀内容如果不够可信被阅读就无法表现。

### 可信度手册：验证、丰富、负责

| 原则 | 说明 | 案例 |
|------|------|------|
| **验证身份** | 在怀疑开始前消除疑虑 | 银行通过验证身份发送一次性密码和欺诈警报 |
| **丰富消息** | 使消息自解释：谁发的、发生了什么、为什么重要、下一步做什么 | 航空公司发送延误通知，包含改签选项、支持联系方式、登机详情 |
| **负责参与** | 偏好优先、频率控制、透明同意、易退出 | 零售品牌让客户设置促销偏好（每周、每月、仅大促） |

### 验证消息优先场景

| 场景类型 | 示例 | 效果 |
|----------|------|------|
| **时间敏感运营时刻** | 配送更新、预约提醒、欺诈警报 | 更少支持工单、更高完成率 |
| **高意图电商时刻** | 购物车放弃、补货提醒、降价通知 | 更高点击率和购买率 |
| **账户和身份时刻** | 登录码、密码重置、账户更改 | 更高成功率、敏感流程更少摩擦 |
| **客户关怀时刻** | 问题解决、状态更新 | 更快解决、摩擦期间更低流失 |
| **生命周期入门** | 偏好、档案完成、首价值时刻 | 更高激活、更强长期参与 |

### 如何像增长团队一样测量信任

| 指标 | 问题 |
|------|------|
| **参与度提升** | 运营消息是否快速打开无需后续检查？ |
| **转化提升** | 购物车恢复点击是否转化为结账完成？ |
| **退出率** | 营销活动后激增=频率或可信度问题信号 |
| **投诉和垃圾报告** | 上升=你在训练客户不信任你 |
| **支持避免** | 更少"这是真的吗？"工单=验证在工作 |
| **长期影响** | 复购、流失、终身价值 |

### 验证消息推广三阶段

```
Phase 1: 保护最高风险、最高价值流程
├── 身份验证、账单、配送、账户更改
└── 示例：金融科技首先验证登录码和支付失败

Phase 2: 扩展到收入时刻
├── 购物车恢复、补货、促销（有护栏）
└── 示例：订阅品牌在添加促销优惠前引入验证续订提醒

Phase 3: 跨渠道标准化身份
├── 将相同身份和偏好模型带到邮件、推送、应用内
└── 示例：市场跨渠道对齐消息命名和偏好设置
```

### 基础设施检查清单

- [ ] 同意已捕获并存储
- [ ] 偏好中心正常工作
- [ ] 跨渠道身份一致性
- [ ] 验证不可用时的回退路径
- [ ] 监控和治理

## 关键洞察总结

### 对话式搜索核心认知

> **"对话式搜索不是附加功能，而是替代摩擦的体验层。"**

- 赢家不会为AI而AI，他们将消除摩擦、建立信心、帮助客户以清晰和尊重完成他们开始的事
- 每次互动都感觉像从理解你约束和目标的人那里获得建议
- 从小开始：本周审计你的最高发现时刻，问一个问题：哪里可以用对话替代迷宫？

### 验证消息核心认知

> **"在发送消息便宜的世界里，可信度是昂贵的。"**

- 验证消息是：
  - **品牌保护**：减少冒充损害
  - **客户体验**：消除高赌注时刻的摩擦
  - **增长**：大规模解锁注意力和转化
- 早期行动的品牌将获得持久优势

### 新增长方程

```
验证身份 + 尊重体验 = 持久注意

下一个参与时代不会由发送更多消息的人赢得
→ 将由赢得被相信权利的人赢得
```

## 实践建议

### 对话式搜索启动建议

1. **从小开始**：选择一个高价值试点旅程
2. **关注完成**：优化完成率，而非点击
3. **设计对话**：时机、语气、问题顺序影响结果
4. **测量正确指标**：完成率、解决时间、满意度
5. **建立信任边界**：明确什么可自动化、什么需人工

### 验证消息启动建议

1. **从最高风险开始**：身份验证、账单、配送
2. **跨渠道一致**：相同身份模型贯穿所有渠道
3. **透明化偏好**：让用户控制他们接收什么
4. **测量信任指标**：参与度、转化、退出率、投诉
5. **建立基础设施**：同意管理、偏好中心、监控治理
