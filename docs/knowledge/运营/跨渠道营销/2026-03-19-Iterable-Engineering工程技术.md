## 二、API 集成体系

### 1. 核心 API 端点

| API 类别 | 端点前缀 | 功能 | 典型用途 |
|----------|----------|------|----------|
| **Users** | `/users` | 用户管理 | 创建、更新、删除用户 |
| **Events** | `/events` | 事件追踪 | 行为数据采集 |
| **Campaigns** | `/campaigns` | 营销活动 | 触发、管理营销 |
| **Commerce** | `/commerce` | 电商数据 | 订单、购物车 |
| **Catalogs** | `/catalogs` | 产品目录 | 动态内容 |
| **In-App** | `/inApp` | 应用内消息 | 消息管理 |
| **Push** | `/push` | 推送通知 | 设备管理 |

### 2. 事件追踪最佳实践

```javascript
// 标准事件结构
{
  "email": "user@example.com",
  "userId": "user123",
  "eventName": "purchase_completed",
  "dataFields": {
    "productId": "prod_001",
    "productName": "Premium Plan",
    "price": 99.99,
    "currency": "USD",
    "category": "subscription"
  },
  "createdAt": "2026-03-19T10:30:00Z"
}
```

### 3. 事件命名规范

| 原则 | 好的命名 | 不好的命名 |
|------|----------|-----------|
| 使用动词 | `purchase_completed` | `purchase` |
| 蛇形命名 | `product_viewed` | `productViewed` |
| 语义清晰 | `subscription_renewed` | `sub_renew` |
| 层级结构 | `order.item.added` | `addItem` |

## 三、数据管道设计

### 1. 数据流向架构

```
数据源 → 数据采集 → 数据处理 → Iterable → 消息发送
    │         │          │          │          │
    ▼         ▼          ▼          ▼          ▼
 Web/App   SDK/API    ETL/CDP   工作流    渠道适配
 CRM       Webhooks   数据清洗   分段      Email/SMS
 CDP       批量导入   身份解析   个性化    Push/InApp
```

### 2. 实时 vs 批量处理

| 场景 | 推荐方式 | 延迟 | 示例 |
|------|----------|------|------|
| 行为触发 | 实时事件 | <1秒 | 购买后立即发确认 |
| 分段更新 | 批量同步 | 15分钟-1小时 | VIP用户每日更新 |
| 大规模导入 | 批量API | 数小时 | 历史数据迁移 |
| 实时个性化 | 实时API | <100ms | 动态内容渲染 |

### 3. 数据质量保障

```
数据质量检查清单：
    ├─ 完整性：必填字段是否存在
    ├─ 准确性：数据格式是否正确
    ├─ 一致性：跨系统数据是否一致
    ├─ 时效性：数据是否及时更新
    └─ 唯一性：是否存在重复记录
```

## 四、身份识别与合并

### 1. 身份识别策略

```
匿名用户 → 已知用户 → 统一身份
    │          │          │
    ▼          ▼          ▼
设备ID     Email/ID    跨设备识别
临时标识   登录关联    行为归因
```

### 2. Iterable 身份模型

| 标识类型 | 字段 | 用途 | 优先级 |
|----------|------|------|--------|
| **email** | 主键 | 主要标识 | 最高 |
| **userId** | 自定义ID | 系统集成 | 高 |
| **deviceId** | 设备标识 | 移动端追踪 | 中 |
| **iterableCampaignId** | 活动ID | 归因追踪 | 低 |

### 3. 身份合并最佳实践

```javascript
// 推荐的身份合并流程
// Step 1: 匿名阶段 - 使用临时ID
identify({ deviceId: 'device_123' });

// Step 2: 注册/登录 - 关联邮箱
identify({ email: 'user@example.com', deviceId: 'device_123' });

// Step 3: 确保数据合并
// Iterable 自动合并相同 email 的用户数据
```

## 五、工作流与自动化

### 1. 工作流类型

| 类型 | 触发方式 | 用途 | 复杂度 |
|------|----------|------|--------|
| **触发式** | 事件触发 | 即时响应 | ★ |
| **定时式** | 时间触发 | 定期任务 | ★★ |
| **API触发** | API调用 | 系统集成 | ★★ |
| **复杂旅程** | 多条件组合 | 生命周期 | ★★★ |

### 2. 工作流设计模式

```
模式1: 简单触发
事件 → 消息

模式2: 条件分支
事件 → 条件判断 → 分支A消息
                   └→ 分支B消息

模式3: 延迟等待
事件 → 等待N时间 → 检查条件 → 消息

模式4: 多渠道编排
事件 → 渠道A → 等待 → 渠道B → 等待 → 渠道C
```

### 3. 错误处理策略

| 错误类型 | 处理策略 | 代码示例 |
|----------|----------|----------|
| 发送失败 | 重试机制 | `retry(3, 60s)` |
| 用户不存在 | 创建或跳过 | `upsert: true` |
| 模板错误 | 降级模板 | `fallback_template` |
| 渠道限制 | 备用渠道 | `fallback_channel` |

## 六、性能优化

### 1. API 调用优化

| 优化策略 | 描述 | 效果 |
|----------|------|------|
| **批量操作** | 合并多个请求 | 减少API调用次数 |
| **异步处理** | 非阻塞调用 | 提升响应速度 |
| **缓存利用** | 本地缓存数据 | 减少重复请求 |
| **限流控制** | 遵守速率限制 | 避免被拒绝 |

### 2. 批量API示例

```javascript
// 批量更新用户
POST /api/users/bulkUpdate
{
  "users": [
    { "email": "user1@example.com", "dataFields": { "tier": "gold" } },
    { "email": "user2@example.com", "dataFields": { "tier": "silver" } },
    { "email": "user3@example.com", "dataFields": { "tier": "gold" } }
  ]
}
```

### 3. 性能基准

| 操作类型 | 建议批量大小 | 速率限制 |
|----------|--------------|----------|
| 用户更新 | 100-1000/批 | 10000/分钟 |
| 事件追踪 | 100-500/批 | 50000/分钟 |
| 消息触发 | 100-1000/批 | 因渠道而异 |

## 七、安全与合规

### 1. 认证方式

| 方式 | 用途 | 安全级别 |
|------|------|----------|
| **API Key** | 服务端集成 | 高（需保护） |
| **JWT** | 移动端集成 | 中（有过期） |
| **OAuth** | 第三方授权 | 高（标准协议） |

### 2. 数据安全最佳实践

```
安全检查清单：
    ├─ API Key 不暴露在前端代码
    ├─ 使用环境变量存储敏感信息
    ├─ 实施IP白名单（如可能）
    ├─ 定期轮换API Key
    ├─ 最小权限原则配置
    └─ 日志中不记录敏感数据
```

### 3. 合规要求

| 法规 | 地区 | 关键要求 |
|------|------|----------|
| **GDPR** | 欧盟 | 同意管理、数据删除权 |
| **CCPA** | 加州 | 选择退出、数据访问权 |
| **CAN-SPAM** | 美国 | 退订机制、真实发件人 |
| **CASL** | 加拿大 | 明确同意、身份识别 |

## 八、监控与调试

### 1. 监控指标

| 指标类别 | 具体指标 | 告警阈值 |
|----------|----------|----------|
| **发送性能** | 发送成功率、延迟 | 成功率<95% |

## 九、集成案例

### 1. CDP 集成架构

```
数据源 → CDP(mParticle/Segment) → Iterable → 消息渠道
              │
              ├─ 用户数据同步
              ├─ 事件流转发
              └─ 分段数据共享
```

### 2. 电商平台集成

```
电商平台 → Iterable
    │
    ├─ 用户注册/登录 → 用户创建
    ├─ 浏览行为 → 事件追踪
    ├─ 购物车 → 购物车事件
    ├─ 订单 → 交易事件
    └─ 商品目录 → Catalog同步
```

### 3. 移动应用集成

```
移动App → Iterable SDK → Iterable
    │
    ├─ 设备注册 → Push Token
    ├─ 用户行为 → 自定义事件
    ├─ In-App消息 → 消息展示
    └─ 深度链接 → 跳转处理
```

## 十、关键洞察

1. **API设计优先**: 集成前规划好事件结构和数据模型
2. **批量优于单个**: 大规模操作使用批量API
3. **错误处理关键**: 健壮的错误处理机制是必需的
4. **监控不可少**: 建立完善的监控告警体系
5. **文档即代码**: 维护清晰的API文档和集成说明
