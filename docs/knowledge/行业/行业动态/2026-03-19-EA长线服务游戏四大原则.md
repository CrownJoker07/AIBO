## 核心框架：更新内容的三重视角

每次更新必须通过三个视角审视：

```
┌─────────────────────────────────────────────────────────┐
│                    三重视角框架                          │
├─────────────────────────────────────────────────────────┤
│  👥 Players (玩家)                                      │
│     └── 新内容如何激励玩家持续参与                       │
│                                                         │
│  👨‍👩‍👧‍👦 People (团队)                                     │
│     └── 团队是否理解每次更新的目标和商业背景             │
│                                                         │
│  ⚙️ Process (流程)                                      │
│     └── 如何以最高效的方式创建新内容                    │
└─────────────────────────────────────────────────────────┘
```

## 四大常青原则

### 原则 1: 随时准备改变，无论投入多少

**核心观点**: 保持灵活性，根据实际玩家行为调整，而非固守计划

**案例: Bejeweled Blitz "Encore" 功能**
- PopCap 团队花费 **数月** 开发新功能 "Encore"
- 功能: 玩家可花费金币提升分数
- 问题: 焦点小组测试发现，玩家只是点击 "X" 关闭弹窗
- 教训: 即使投入大量工作，也要敢于根据反馈重新设计

```
开发流程调整:
原计划: 设计 → 开发 → 微调UI → 发布
↓
实际需要: 设计 → 开发 → 测试 → 观察玩家行为 → 重新设计 → 发布
```

**关键洞察**:
> "玩家只是点击 X 关闭弹窗，他们唯一关心的是分数提升。重新设计后效果显著提升。"

---

### 原则 2: 给玩家想要的东西

**核心观点**: 聚焦玩家真正关心的内容，而非你认为重要的内容

**案例: Plants vs Zombies 2 内容策略转型**

| 阶段 | 策略 | 周期 | 内容 | 价格 |
|------|------|------|------|------|
| 初期 | 世界扩展 | 2-3个月 | 新关卡 + 20-30种植物 + 僵尸 + 挑战 | $5/世界 |
| 转型后 | 高级植物 | 每月 | 单一特色植物 + 相关任务/事件 | $5/植物 |

**转型原因**:
- 发现玩家最享受的是 **新植物的独特性**
- 将资源从"重内容跑步机"转向玩家真正关心的内容

```
资源分配变化:
初期:
├── 世界设计 (40%)
├── 关卡制作 (30%)
├── 植物设计 (20%)
└── 其他内容 (10%)

转型后:
├── 植物设计 (60%)
├── 相关任务/事件 (30%)
└── 其他内容 (10%)
```

---

### 原则 3: 将更新仪式化

**核心观点**: 建立规律的更新节奏，让游戏成为玩家日常的一部分

**更新节奏类型**:

| 类型 | 周期 | 作用 |
|------|------|------|
| 中期更新 | 2-3个月 | 大型扩展包 |
| 每日更新 | 每天 | 日常任务 |
| 定时提醒 | 实时 | 建立期待感 |

**案例: Plants vs Zombies 2 每日任务计时器**

```
DAU 影响分析:
├── 添加计时器前: DAU 波动大
├── 添加计时器后: DAU 稳定化
└── 新任务上线时: DAU 明显飙升
```

**关键洞察**:
- 玩家知道何时有新内容 → 可以规划游戏时间
- 预告/计时器 → 建立期待感 → 提升参与度

---

### 原则 4: 玩家更关心得到什么，而非你如何制作

**核心观点**: 关注玩家感知的价值，而非开发技术细节

**玩家关心的"三个How"（按重要性排序）**:

```
重要性排序:
┌────────────────────────────────────────────────────┐
│  1️⃣ How Often (更新频率)     ⭐⭐⭐⭐⭐            │
│     └── 你多频繁地与他们互动                        │
│                                                    │
│  2️⃣ How Much (内容量)        ⭐⭐⭐⭐              │
│     └── 你提供多少新内容                           │
│                                                    │
│  3️⃣ How (制作方式)           ⭐                   │
│     └── 技术实现细节 (最不重要)                     │
└────────────────────────────────────────────────────┘
```

**玩家不在乎的技术细节**:
- 是换皮还是从头开发？
- 是客户端更新还是服务器推送？
- 使用什么技术栈？

**玩家只想要**: 新的、有趣的内容来互动

## 实践清单

### 更新规划时
- [ ] 是否从玩家视角审视了内容价值？
- [ ] 团队是否理解这次更新的商业目标？
- [ ] 制作流程是否足够高效？

### 发布后
- [ ] 观察玩家实际行为，而非假设
- [ ] 准备根据反馈调整计划
- [ ] 关注玩家真正享受的内容部分

### 长期运营
- [ ] 建立规律的更新节奏
- [ ] 用预告/计时器建立期待感
- [ ] 聚焦"更新频率"和"内容量"，而非技术炫技

## 与其他知识的关联

### 相关案例
- **Scopely 长期主义**: 7年开发Monopoly Go，持续迭代
- **Kingshot 快速增长**: 首年收入$8亿，但需维护核心付费玩家

### 设计哲学关联
- **留存优先**: 行业趋势从获客转向留存
- **玩家中心**: 所有更新以玩家体验为出发点
