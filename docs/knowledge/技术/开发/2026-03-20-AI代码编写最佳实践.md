## 一、语义函数 (Semantic Functions)

### 定义

语义函数是代码库的构建块，应该**尽可能精简**以**优先保证正确性**。

### 核心原则

| 原则 | 说明 |
|------|------|
| **最小化** | 函数应尽可能小 |
| **自描述** | 代码本身即是定义，无需注释 |
| **无副作用** | 除非副作用是明确目标 |
| **可测试** | 好的语义函数极易单元测试 |

### 输入输出要求

- 接收完成目标所需的所有输入
- 直接返回所有必要输出
- 可包装其他语义函数来描述流程

### 命名示例

```
quadratic_formula()
retry_with_exponential_backoff_and_run_y_in_between<Y: func, X: Func>(x: X, y: Y)
```

即使这些函数不再被使用，未来的人类和代理也会欣赏这种信息索引。

### 副作用处理

**一般不鼓励副作用**，因为语义函数应该可以安全复用而无需了解其内部实现。

**复杂逻辑处理模式**：
- 将复杂流程拆分为一系列自描述的语义函数
- 每个函数只接收所需输入
- 返回下一步所需数据
- 不做其他任何事情

## 二、实用函数 (Pragmatic Functions)

### 定义

实用函数是语义函数和独特逻辑的包装器，是代码库中的**复杂流程**。

### 核心原则

| 原则 | 说明 |
|------|------|
| **包装器** | 包装语义函数和独特逻辑 |
| **有限复用** | 一般不应在多处使用 |
| **集成测试** | 测试属于集成测试范畴 |
| **预期变化** | 内部和功能都会随时间完全改变 |

### 命名示例

```
provision_new_workspace_for_github_repo(repo, user)
handle_user_signup_webhook()
```

### 文档注释指南

**避免**：
- 重复函数名称
- 描述显而易见的特征

**应该记录**：
- 意外行为（如"balance 小于 10 时提前失败"）
- 消除来自函数名称的误解

**阅读提醒**：
> 文档注释需要带着怀疑态度阅读，函数内部的编码者可能忘记更新它们

### 重构信号

如果实用函数在多处使用，考虑：
1. 拆分显式逻辑
2. 移动到语义函数中

## 三、数据模型 (Models)

### 核心原则

**数据形状应使错误状态不可能存在**

### 问题信号

| 问题 | 后果 |
|------|------|
| 可选字段过多 | 代码库每次接触数据都要回答问题 |
| 类型过于宽松 | 调用者可能传入看起来正确但实际错误的数据 |
| 字段组合不应存在 | 模型没有完成其职责 |

### 命名精确性

模型名称应足够精确，以至于：
- 可以查看任何字段并知道它是否属于该模型
- 如果名称不能告诉你，说明模型试图做太多事情

### 组合优于合并

```typescript
// 好：保持两个模型完整
UserAndWorkspace {
  user: User,
  workspace: Workspace
}

// 差：将工作区字段扁平化到用户中
```

### 好的命名示例

- `UnverifiedEmail` - 告诉你确切什么字段属于
- `PendingInvite` - 清晰的领域概念
- `BillingAddress` - 如果看到 `phone_number` 字段，就知道出问题了

### 品牌类型 (Brand Types)

**问题**：相同形状的值可能代表完全不同的领域概念

```typescript
{ id: "123" }  // 可能是 DocumentReference 或 MessagePointer
```

**解决方案**：品牌类型包装原语

```typescript
DocumentId(UUID)  // 而不是裸 UUID
MessageId(UUID)
```

**效果**：意外交换两个 ID 会变成**语法错误**，而不是三层深处的静默 bug

## 四、代码腐化点 (Where Things Break)

### 语义函数 → 实用函数的退化

**常见问题**：
- 语义函数为了便利变成实用函数
- 代码库其他依赖它的地方做了非预期的事情

**解决方案**：
- 创建函数时**按使用位置命名**，而非按功能命名
- 名称性质应向其他程序员表明：
  - 行为未紧密定义
  - 内部不应依赖其执行精确任务
  - 便于调试回归问题

### 模型退化（更慢但同样致命）

**退化过程**：
1. 模型开始是专注的
2. 有人添加"仅仅一个更多"的可选字段（因为比创建新模型更容易）
3. 其他人也这样做
4. 最终模型变成松散的半相关数据袋
5. 每个消费者都要猜测哪些字段实际设置以及为什么

**退化信号**：
- 名称停止描述数据是什么
- 字段不再围绕单一概念凝聚
- 每个新功能都要导航模型从未设计代表的状态

**重构时机**：
> 当模型的字段不再围绕其名称凝聚时，就是将其拆分为它一直耦合的不同事物的信号

## 核心洞察

### 1. 自文档化代码

代码应该自描述，语义函数不需要任何注释。

### 2. 类型安全

- 使错误状态在类型层面不可能
- 使用品牌类型区分相同形状的不同概念
- 优先组合而非合并

### 3. 函数分类

| 类型 | 用途 | 测试方式 | 预期寿命 |
|------|------|----------|----------|
| 语义函数 | 构建块 | 单元测试 | 长期稳定 |
| 实用函数 | 复杂流程 | 集成测试 | 预期变化 |

### 4. 防腐化策略

- 语义函数按功能命名
- 实用函数按使用位置命名
- 模型字段与名称保持凝聚
- 可选字段是代码异味
