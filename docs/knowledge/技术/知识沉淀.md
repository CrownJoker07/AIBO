### 游戏开发

#### 游戏编程模式

**六大类别**：

| 类别 | 模式 | 用途 |
|------|------|------|
| **设计模式重访** | Command, Flyweight, Observer, Prototype, Singleton, State | GoF 模式在游戏中的应用 |
| **时序模式** | Double Buffer, Game Loop, Update Method | 控制游戏时间流动 |
| **行为模式** | Bytecode, Subclass Sandbox, Type Object | 定义实体行为 |
| **解耦模式** | Component, Event Queue, Service Locator | 降低代码依赖 |
| **优化模式** | Data Locality, Dirty Flag, Object Pool, Spatial Partition | 提升性能 |

**关键设计原则**：
1. **架构 vs 性能**：好的架构通常意味着更好的性能可预测性
2. **解耦的重要性**：降低代码间的依赖关系，使修改和扩展更容易
3. **游戏特有的考量**：实时性要求、资源管理、状态管理

#### Unity 渲染与着色器编程

**变换矩阵核心**：

| 矩阵 | 用途 |
|------|------|
| `unity_ObjectToWorld` | 模型→世界空间 |
| `UNITY_MATRIX_V` | 世界→视图空间 |
| `UNITY_MATRIX_P` | 视图→裁剪空间 |
| `UNITY_MATRIX_IT_MV` | 法线变换（逆转置） |

**齐次坐标**：4D 点 `(x,y,z,w)`，w=1 为点，w=0 为方向

**着色器基础结构**：
```hlsl
// 顶点着色器：模型空间 → 裁剪空间
i.position = mul(UNITY_MATRIX_MVP, v.position);
// UV 变换：Tiling + Offset
i.uv = TRANSFORM_TEX(v.uv, _MainTex);
// 片段着色器：纹理采样 × 颜色
return tex2D(_MainTex, i.uv) * _Tint;
```

**纹理过滤模式**：
| 模式 | 适用场景 |
|------|----------|
| Point | 像素艺术 |
| Bilinear | 2D 游戏 |
| Trilinear | 3D 场景（Mipmap） |
| Anisotropic | 斜角纹理（地面） |

**自定义 SRP 渲染顺序**：Opaque → Skybox → Transparent

**六边形网格**：
- 外径/内径比：inner = outer × 0.866 (√3/2)
- Cube 坐标：x + y + z = 0

**噪声算法对比**：
| 类型 | 特点 | 用途 |
|------|------|------|
| Perlin | 梯度插值，自然连续 | 地形、云 |
| Simplex | 单纯形网格，更少伪影 | 高维噪声 |
| Worley | 细胞状图案 | 水面、石材 |

#### Unreal Engine 5 核心技术

**五大核心系统**：

| 系统 | 用途 | 关键特性 |
|------|------|----------|
| **Nanite** | 虚拟化几何 | 像素级精度、自动LOD、无限多边形 |
| **Lumen** | 全局光照 | 实时GI、动态反射、自发光照明 |
| **VSMs** | 阴影贴图 | 16k分辨率、SMRT软阴影、自动管理 |
| **World Partition** | 世界管理 | 空间分区、距离流式加载、数据层 |
| **Blueprints** | 可视化脚本 | 节点式编程、事件驱动、快速原型 |

**Nanite 技术要点**：
- 只渲染屏幕可见三角形（像素级精度）
- 智能自动 LOD，无需手动创建
- 支持静态网格、骨骼网格（有限）、地形、植被
- 限制：不支持透明材质、世界位置偏移

**Lumen 渲染技术**：
- 表面缓存（Surface Cache）加速间接光照
- 硬件光线追踪（RTX）+ 软件光线追踪
- 高质量模式需要 RTX，性能模式适用更多平台

**Virtual Shadow Maps**：
- 最高 16k x 16k 分辨率
- SMRT（屏幕空间阴影光线追踪）提供软阴影
- 智能缓存策略，按需更新

**World Partition 架构**：
```
World Partition
├── 空间分区 → 运行时单元格
├── 数据层 → 逻辑分组
└── HLOD → 层级细节优化
```

**One File Per Actor**：每个 Actor 独立文件，支持 World Partition，优化团队协作

| 原型开发 | Blueprint |
| 玩法逻辑 | Blueprint |

### AI 代码编写

#### AI Code 最佳实践

**核心理念**：有意识地控制 AI 编写的代码质量

**函数分类**：
| 类型 | 用途 | 测试方式 | 命名方式 |
|------|------|----------|----------|
| 语义函数 | 构建块 | 单元测试 | 按功能命名 |
| 实用函数 | 复杂流程 | 集成测试 | 按使用位置命名 |

**语义函数原则**：
- 尽可能精简，优先保证正确性
- 自描述，无需注释
- 一般不鼓励副作用
- 极易单元测试

**实用函数原则**：
- 语义函数和独特逻辑的包装器
- 预期随时间完全改变
- 文档注释记录意外行为，而非重复函数名

**数据模型设计**：
- 数据形状应使错误状态不可能存在
- 可选字段是代码异味
- 组合优于合并（UserAndWorkspace 而非扁平化）
- 使用品牌类型区分相同形状的不同概念

**代码腐化信号**：
- 语义函数变成实用函数
- 模型字段不再围绕名称凝聚
- 名称停止描述数据是什么

**品牌类型示例**：
```typescript
DocumentId(UUID)  // 而不是裸 UUID
MessageId(UUID)   // 交换 ID 变成语法错误
```

**t-SNE 可视化六大陷阱**：
| 陷阱 | 说明 |
|------|------|
| 超参数很重要 | perplexity 应在 5-50 范围 |
| 聚类大小无意义 | 算法会自然扩展/收缩聚类 |
| 聚类间距离可能无意义 | 需要精细调整 perplexity |
| 随机噪声不总是随机 | 低 perplexity 会产生"虚假"聚类 |
| 形状可见但需正确 perplexity | 高 perplexity 适合 elongated shapes |
| 拓扑需要多图 | 多个 perplexity 值给出最完整图景 |

**特征可视化核心**：
| 优化目标 | 公式 | 说明 |
|----------|------|------|
| 神经元 | `layer_n[x,y,z]` | 单个位置神经元 |
| 通道 | `layer_n[:,:,z]` | 整个通道 |
| 层/DeepDream | `layer_n[:,:,:]²` | 整层"有趣"图像 |
| 类别 Logits | `pre_softmax[k]` | softmax 前类别证据 |

**特征可视化三大敌人**：
1. **高频噪声** - 步进卷积和池化操作产生
2. **缺乏多样性** - 单一优化只显示特征"一个切面"
3. **神经元交互** - 组合效应难以理解

**正则化谱系**：
| 方法 | 说明 |
|------|------|
| 频率惩罚 | 总变分、模糊、双边滤波 |
| 变换鲁棒性 | 抖动、旋转、缩放 |
| 学习先验 | GAN、VAE、去噪自编码器 |

**动量优化核心洞察**：
```
条件数 κ = λ_n / λ_1

梯度下降收敛率: (κ-1)/(κ+1)
动量收敛率:     (√κ-1)/(√κ+1)
```

💡 **动量本质是将条件数开平方！**

**动量最优参数**：
```
最优步长 α = (2/(√λ_1 + √λ_n))²
最优动量 β = ((√λ_n - √λ_1)/(√λ_n + √λ_1))²
```

**动量实用指南**：
- β 接近 1 (0.99 或 0.999)
- α 约为梯度下降最优值的两倍
- 在收敛边缘是好事
- 随机情况下注意噪声影响

#### Claude Code 浏览器自动化方案

**五大方案对比**：

| 方案 | Token消耗 | 核心优势 | 适用场景 |
|------|-----------|----------|----------|
| **Browser-use** | 极低 | 多模式+会话持久+云端并行 | 登录态/并行采集/反爬绕过 |
| **Agent Browser** | 极低(减少93%) | 快、省Token | 快速浏览、截图、简单操作 |
| **Playwright CLI** | 极低(减少75-99%) | 省Token+跨浏览器 | 测试和自动化（有Shell权限） |
| **Playwright MCP** | 较高 | 稳定、功能全 | 沙盒环境中做浏览器自动化 |
| **DevTools MCP** | 中等 | 调试能力强 | 调试排错、查看网络请求 |

**Token 节省效果**：
- 单个页面快照: ~15,000 → ~200 tokens（节省98.7%）
- 10步操作: ~114,000 → ~27,000 tokens（节省76.3%）

**选择口诀**：
- 看看、填表 → Agent Browser
- 登录态、并行、反爬 → Browser-use
- 测试、跑流程（有Shell）→ Playwright CLI
- 测试、跑流程（沙盒）→ Playwright MCP
- 调试、抓请求 → DevTools MCP

#### Kimi 注意力残差技术 (Attention Residuals)

**背景**：传统残差连接（ResNet 2015）的局限
```python
h = h + f(h)  # 等权累加，浅层信息被冲淡
```

**核心创新**：将 Transformer 中用于"时间"维度的注意力机制，旋转 90 度用于"深度"维度

| 技术 | 思路 | 年份 |
|------|------|------|
| ResNet | 一条直通车 | 2015 |
| 字节超连接 | 加三条车道 | 2024 |
| DeepSeek mHC | 装限速器 | 2025底 |
| Kimi AttnRes | 换发动机 | 2026.3 |

**Block AttnRes**：
- 把模型分成块
- 块内：传统累加
- 块间：用注意力选择

**实验结果**：
| 测试 | 提升 |
|------|------|
| GPQA-Diamond (博士级科学问答) | +7.5% |
| Minerva Math (数学推理) | +3.6% |
| HumanEval (代码生成) | +3.1% |

**关键洞察**：
- 算力等效：Block AttnRes 相当于 **白送 1.25 倍算力**
- 越难的任务，提升越大
- 显存开销比 mHC **低六倍**

### 设计模式

#### 23 种设计模式速查

**创建型模式（5个）**：
| 模式 | 用途 | 适用场景 |
|------|------|----------|
| **Factory Method** | 创建对象接口 | 无法预知对象确切类型 |
| **Abstract Factory** | 创建产品系列 | 需要多个产品系列配合 |
| **Builder** | 分步构建复杂对象 | 构造函数参数过多 |
| **Prototype** | 克隆对象 | 对象创建成本高 |
| **Singleton** | 全局唯一实例 | 需要全局访问点（谨慎使用） |

**结构型模式（7个）**：
| 模式 | 用途 | 适用场景 |
|------|------|----------|
| **Adapter** | 接口转换 | 不兼容接口协同工作 |
| **Bridge** | 分离抽象与实现 | 两者需要独立变化 |
| **Composite** | 树形结构 | 部分整体层次结构 |
| **Decorator** | 动态添加职责 | 运行时添加/移除职责 |
| **Facade** | 简化复杂接口 | 为子系统提供简单接口 |
| **Flyweight** | 共享细粒度对象 | 大量相似对象节省内存 |
| **Proxy** | 控制访问 | 延迟初始化/访问控制/缓存 |

**行为型模式（11个）**：
| 模式 | 用途 | 适用场景 |
|------|------|----------|
| **Chain of Responsibility** | 处理链 | 多个对象可处理请求 |
| **Command** | 封装请求 | 撤销/重做、延迟执行 |
| **Iterator** | 遍历集合 | 不暴露底层表示 |
| **Mediator** | 集中通信 | 减少组件间混乱依赖 |
| **Memento** | 保存状态 | 撤销/恢复 |
| **Observer** | 事件通知 | 一对多依赖关系 |
| **State** | 状态相关行为 | 消除大量条件语句 |
| **Strategy** | 可切换算法 | 运行时切换算法 |
| **Template Method** | 算法骨架 | 复用代码结构 |
| **Visitor** | 分离操作和结构 | 添加新操作不修改结构 |

#### SOLID 原则

| 原则 | 名称 | 描述 |
|------|------|------|
| **S** | 单一职责原则 (SRP) | 一个类只负责一件事 |
| **O** | 开闭原则 (OCP) | 对扩展开放，对修改关闭 |
| **L** | 里氏替换原则 (LSP) | 子类可以替换父类 |
| **I** | 接口隔离原则 (ISP) | 不应强迫依赖不使用的方法 |
| **D** | 依赖倒置原则 (DIP) | 依赖抽象，不依赖具体 |

#### 设计模式使用建议

**何时使用**：
1. 有明确问题时才使用
2. 理解权衡：每个模式都有成本
3. 团队共识：确保团队理解所选模式
4. 保持简单：不要过度设计

**常见反模式**：
- 滥用单例：导致全局状态和测试困难
- 过度使用工厂：简单对象不需要工厂
- 装饰器堆叠：难以理解和调试
- 为模式而模式：不需要时强行使用

## 最佳实践

1. **游戏开发**：
   - 使用 Component 模式实现 ECS 架构
   - 对象池减少内存分配和 GC
   - 数据局部性优化缓存命中

2. **Unity 渲染**：
   - GPU Instancing 处理相同材质多物体
   - 渲染顺序：Opaque(前→后) → Skybox → Transparent(后→前)
   - 着色器用 `saturate()` 代替 `clamp(x, 0, 1)`
   - 避免片段着色器中使用 `discard`

3. **浏览器自动化**：
   - 首选 Browser-use（最全面）
   - 侧重编程测试选 Playwright CLI
   - 日常浏览体验加 Agent Browser

4. **设计模式**：
   - 优先使用组合而非继承
   - 针对接口编程，不针对实现编程
   - 不要过度设计，只在真正需要时使用模式

5. **LLM 架构**：
   - 注意力残差可替代传统残差连接
   - Block AttnRes 在保持性能的同时降低显存
   - 越难的任务，架构创新收益越大

6. **AI 学习路径**：
   - 先理解原理（micrograd）→ 再实践应用（nanoGPT）
   - 从零构建比调用 API 更有教育价值
   - 教学是深化理解的最好方式

7. **t-SNE 可视化**：
   - 尝试多个 perplexity 值（5, 30, 50）
   - 确保足够迭代次数（直到稳定）
   - 不要过度解读聚类大小和距离
   - 结合其他方法验证结论

8. **特征可视化**：
   - 使用通道目标而非单神经元
   - 在去相关傅里叶空间优化
   - 添加多样性项和变换鲁棒性
   - 至少 2560 步优化

9. **动量优化**：
   - β 接近 1 (0.99 或 0.999)
   - α 约为梯度下降最优值的两倍
   - 在收敛边缘是好事
   - 注意随机情况下的噪声影响

10. **Unreal Engine 5**：
    - Nanite: 高模资产直接导入，注意材质兼容性
    - Lumen: 根据平台选择质量模式，合理设置自发光强度
    - VSM: 启用阴影缓存，配合级联阴影使用
    - World Partition: 提前规划数据层结构，使用 HLOD 减少远景开销
    - Blueprint: 保持逻辑清晰，性能敏感部分用 C++ 重写

11. **AI 代码编写**：
    - 语义函数按功能命名，实用函数按使用位置命名
    - 数据模型使错误状态不可能存在
    - 使用品牌类型区分相同形状的不同概念
    - 可选字段是代码异味，考虑拆分模型
