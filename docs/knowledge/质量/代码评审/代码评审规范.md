# 代码评审规范

> 通过结构化评审提升代码质量，促进团队知识共享

---

## 评审目标

- 发现潜在 Bug 和安全问题
- 确保代码可读性和可维护性
- 促进团队技术交流
- 保持代码风格一致性

---

## 评审原则

### 评审者

- 及时响应 PR 评审请求（24 小时内）
- 评论建设性、具体、可操作
- 区分 "必须修改" 和 "建议修改"
- 给予肯定和表扬

### 被评审者

- 保持开放心态接受反馈
- 提供清晰的 PR 描述
- 自检后再提交评审
- 逐条回复评审意见

---

## 评审检查项

### 功能正确性

- [ ] 代码逻辑正确
- [ ] 边界条件处理
- [ ] 错误处理完善
- [ ] 单元测试覆盖

### 代码质量

- [ ] 命名清晰有意义
- [ ] 函数职责单一
- [ ] 无重复代码
- [ ] 适当注释

### 性能安全

- [ ] 无性能问题
- [ ] 无安全漏洞
- [ ] 敏感信息不硬编码
- [ ] 数据库查询优化

### 一致性

- [ ] 符合团队代码规范
- [ ] 与现有代码风格一致
- [ ] API 设计合理

---

## PR 描述模板

```markdown
## 背景
为什么需要这个变更？

## 变更内容
- 变更点1
- 变更点2

## 测试情况
- 单元测试: 通过
- 集成测试: 通过
- 手动测试: 已验证

## 影响范围
哪些模块/功能受影响？
```
