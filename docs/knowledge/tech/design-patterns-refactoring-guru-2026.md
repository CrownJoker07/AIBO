## 设计模式概述

设计模式是软件设计中常见问题的典型解决方案。它们是经过时间检验的蓝图，可以帮助解决代码设计中反复出现的问题。

### 模式分类

模式根据其复杂程度、细节层次以及对整个系统的应用范围进行分类。最常见的分类方式是按目的分类：

| 类别 | 描述 | 模式数量 |

| **结构型模式** | 对象组合方式 | 7 个 |
| **行为型模式** | 对象间通信 | 11 个 |

## 一、创建型模式 (Creational Patterns)

创建型模式提供灵活的对象创建机制，增加现有代码的灵活性和复用性。

### 1. Factory Method (工厂方法)

**定义**: 定义一个创建对象的接口，但让子类决定实例化哪个类。

```
Product ←── Creator
    ↑         ↑
ConcreteProduct ←── ConcreteCreator
```

**适用场景**:
- 无法预知对象确切类型
- 让用户扩展库的内部组件
- 复用现有对象而非重建

**优点**: 避免创建者和具体产品之间的紧密耦合 | 符合单一职责原则 | 符合开闭原则

**缺点**: 每增加一个产品都需要增加新的创建者子类

---

### 2. Abstract Factory (抽象工厂)

**定义**: 创建一系列相关或相互依赖对象的接口，无需指定具体类。

```
AbstractFactory ←─── ConcreteFactory1
      ↓                   ↓
AbstractProductA    ProductA1
AbstractProductB    ProductB1
```

**适用场景**:
- 代码需要与多个产品系列配合
- 有一个类的对象库，只暴露接口
- 需要确保产品之间的一致性

**优点**: 产品之间的一致性 | 避免创建者和具体产品之间的紧密耦合 | 符合单一职责原则 | 符合开闭原则

**缺点**: 引入新的产品类型需要修改工厂接口

---

### 3. Builder (建造者)

**定义**: 分离复杂对象的构建和表示，使同样的构建过程可以创建不同的表示。

```
Director → Builder
              ↓
         ConcreteBuilder → Product
```

**适用场景**:
- 构建「有许多配置选项」的复杂对象
- 避免构造函数参数过多（telescoping constructor）
- 分步创建对象

**优点**: 分步创建对象 | 复用相同的构建代码 | 符合单一职责原则

**缺点**: 增加代码复杂度

---

### 4. Prototype (原型)

**定义**: 通过复制已有对象来创建新对象，而不依赖类。

```
Client → Prototype.clone() → 新对象
```

**适用场景**:
- 需要复制对象，同时保持性能
- 对象创建成本高
- 避免继承构造函数的复杂性

**优点**: 复制对象而不耦合 | 避免重复初始化 | 方便生成复杂对象

**缺点**: 克隆包含循环引用的复杂对象可能很困难

---

### 5. Singleton (单例)

**定义**: 确保一个类只有一个实例，并提供全局访问点。

```
Singleton
  - instance
  + getInstance()
```

**适用场景**:
- 必须严格控制的共享资源（如数据库连接）
- 需要全局访问点

**优点**: 保证唯一实例 | 全局访问点 | 延迟初始化

**缺点**: 违反单一职责原则 | 可能导致全局状态 | 多线程问题 | 难以测试

---

## 二、结构型模式 (Structural Patterns)

结构型模式解释如何将对象和类组装成更大的结构，同时保持结构的灵活和高效。

### 1. Adapter (适配器)

**定义**: 使不兼容的接口能够协同工作。

```
Client → Target ← Adapter → Adaptee
```

**两种实现方式**:
- 对象适配器（组合）
- 类适配器（继承）

**适用场景**:
- 使用现有类，但接口不兼容
- 需要复用多个类

**优点**: 符合单一职责原则 | 符合开闭原则

---

### 2. Bridge (桥接)

**定义**: 将抽象与实现分离，使两者可以独立变化。

```
Abstraction ←→ Implementation
     ↑              ↑
RefinedAbstraction  ConcreteImpl
```

**适用场景**:
- 避免抽象和实现之间的永久绑定
- 抽象和实现都需要独立扩展

**优点**: 独立扩展 | 符合开闭原则 | 符合单一职责原则

---

### 3. Composite (组合)

**定义**: 将对象组合成树形结构来表示"部分-整体"的层次结构。

```
Component
    ↑
Leaf ←→ Composite
```

**适用场景**:
- 需要实现树形对象结构
- 希望客户端统一处理简单和复杂元素

**优点**: 统一处理简单和复杂对象 | 符合开闭原则

---

### 4. Decorator (装饰器)

**定义**: 动态地给对象添加额外的职责。

```
Component
    ↑
ConcreteComponent ← Decorator
                       ↑
                   ConcreteDecorator
```

**适用场景**:
- 需要在运行时添加或移除对象职责
- 不能通过继承来扩展对象行为

**优点**: 运行时添加职责 | 单一职责原则 | 可组合多个装饰器

**缺点**: 难以移除特定的装饰器 | 装饰器顺序很重要

---

### 5. Facade (外观)

**定义**: 为复杂的子系统提供一个简单的接口。

```
Client → Facade → Subsystem1
                → Subsystem2
                → Subsystem3
```

**适用场景**:
- 需要为复杂子系统提供简单接口
- 需要将子系统分层

**优点**: 隔离代码复杂性 | 符合单一职责原则 | 符合开闭原则

---

### 6. Flyweight (享元)

**定义**: 通过共享多个对象共有的部分状态来减少内存使用。

```
FlyweightFactory → FlyweightPool
                        ↓
                   Shared Intrinsic State
                   + Unshared Extrinsic State
```

**关键概念**:
- 内在状态 (Intrinsic): 可共享的
- 外在状态 (Extrinsic): 不可共享的

**适用场景**:
- 需要大量相似对象
- 对象的大部分状态可以外部化

**优点**: 节省内存

**缺点**: 可能增加代码复杂度

---

### 7. Proxy (代理)

**定义**: 为另一个对象提供占位符或替身以控制访问。

**代理类型**:
- 虚拟代理 (Virtual): 延迟创建昂贵对象
- 保护代理 (Protection): 控制访问权限
- 远程代理 (Remote): 代表远程对象
- 缓存代理 (Cache): 缓存结果
- 智能引用 (Smart Reference): 附加操作

**适用场景**:
- 延迟初始化
- 访问控制
- 本地执行远程服务
- 记录日志
- 缓存请求结果

**优点**: 控制服务对象 | 管理生命周期 | 在不修改对象的情况下工作

## 三、行为型模式 (Behavioral Patterns)

行为型模式负责对象间的高效沟通和职责分配。

### 1. Chain of Responsibility (责任链)

**定义**: 让请求沿着处理链传递，直到被处理。

```
Handler1 → Handler2 → Handler3 → ...
```

**适用场景**:
- 多个对象可以处理请求
- 希望多个处理者自动处理请求
- 处理顺序需要动态改变

**优点**: 控制处理顺序 | 符合单一职责原则 | 符合开闭原则

---

### 2. Command (命令)

**定义**: 将请求转换为独立对象，包含请求的所有信息。

```
Invoker → Command → Receiver
```

**适用场景**:
- 参数化对象执行的操作
- 将操作放入队列、安排执行
- 实现撤销/重做

**优点**: 单一职责原则 | 开闭原则 | 可撤销 | 延迟执行 | 组合命令

---

### 3. Iterator (迭代器)

**定义**: 提供遍历集合元素的方法，不暴露底层表示。

```
Iterator → Aggregate
```

**适用场景**:
- 需要遍历复杂的数据结构
- 希望同一集合有多种遍历方式
- 统一不同数据结构的遍历接口

**优点**: 符合单一职责原则 | 符合开闭原则 | 支持并行遍历

---

### 4. Mediator (中介者)

**定义**: 减少对象之间的混乱依赖，将通信集中到中介者。

```
Component1 ←→ Mediator ←→ Component2
```

**适用场景**:
- 对象之间存在复杂引用关系
- 难以复用某些组件

**优点**: 减少组件依赖 | 符合单一职责原则 | 符合开闭原则

**缺点**: 中介者可能成为"上帝对象"

---

### 5. Memento (备忘录)

**定义**: 保存和恢复对象的内部状态，不破坏封装。

```
Originator ←→ Memento
    ↓
 Caretaker
```

**适用场景**:
- 需要保存对象状态快照
- 直接访问会暴露实现细节

**优点**: 不破坏封装 | 简化原发器代码

**缺点**: 可能消耗大量内存

---

### 6. Observer (观察者)

**定义**: 定义一对多的依赖关系，当一个对象改变状态时，所有依赖者都会收到通知。

```
Subject ←→ Observer1
        ←→ Observer2
```

**适用场景**:
- 一个对象状态改变需要通知其他对象
- 对象需要观察另一个对象

**优点**: 开闭原则 | 运行时建立关系

**缺点**: 通知顺序不确定 | 大量观察者可能导致性能问题

---

### 7. State (状态)

**定义**: 允许对象在内部状态改变时改变行为。

```
Context → State
   ↑        ↑
ConcreteState1, ConcreteState2...
```

**适用场景**:
- 对象行为取决于状态
- 有大量条件判断（根据状态）

**优点**: 开闭原则 | 单一职责原则 | 消除大量条件语句

---

### 8. Strategy (策略)

**定义**: 定义一系列算法，把它们封装起来，并使它们可以互相替换。

```
Context → Strategy
   ↑        ↑
ConcreteStrategy1, ConcreteStrategy2...
```

**适用场景**:
- 需要在运行时切换算法
- 有很多相似但不同的类
- 算法应该对客户端隐藏

**优点**: 开闭原则 | 避免条件语句 | 运行时切换

---

### 9. Template Method (模板方法)

**定义**: 在超类中定义算法骨架，让子类重写特定步骤。

```
AbstractClass
  + templateMethod()  // 固定流程
  # primitiveOperation1()  // 子类实现
  # primitiveOperation2()
```

**适用场景**:
- 有多个类做几乎相同的事情
- 算法复杂但步骤相同

**优点**: 复用代码 | 符合开闭原则

---

### 10. Visitor (访问者)

**定义**: 将算法与对象结构分离，可以在不修改结构的情况下添加新操作。

```
Visitor → Element
   ↑
ConcreteVisitor1, ConcreteVisitor2...
```

**适用场景**:
- 需要对复杂对象结构的所有元素执行操作
- 需要清理辅助行为
- 业务规则只适用于特定对象

**优点**: 开闭原则 | 单一职责原则 | 可与迭代器配合

**缺点**: 添加新元素类型困难

## 设计原则

### SOLID 原则

| 原则 | 名称 | 描述 |
|------|------|------|
| **S** | 单一职责原则 (SRP) | 一个类只负责一件事 |
| **O** | 开闭原则 (OCP) | 对扩展开放，对修改关闭 |
| **L** | 里氏替换原则 (LSP) | 子类可以替换父类 |
| **I** | 接口隔离原则 (ISP) | 不应强迫依赖不使用的方法 |
| **D** | 依赖倒置原则 (DIP) | 依赖抽象，不依赖具体 |

## 使用建议

### 何时使用设计模式

1. **有明确问题时**: 只有在真正需要时才使用
2. **理解权衡**: 每个模式都有成本
3. **团队共识**: 确保团队理解所选模式
4. **保持简单**: 不要过度设计

### 常见反模式

- **滥用单例**: 导致全局状态和测试困难
- **过度使用工厂**: 简单对象不需要工厂
- **装饰器堆叠**: 难以理解和调试
- **为模式而模式**: 不需要时强行使用
